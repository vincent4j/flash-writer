# Flash Writer 闪电写手

Flash Writer 是一套以“写作思路”为事实源的 AI 长文协作工作流，适用于文章、教程、演讲稿、产品介绍等结构化内容。

它不是输入一个标题就直接生成全文的一次性写作器。作者提供思路、观点、引用资料和最终判断，AI 负责写成完整正文；作者只需 review 和纠偏，不需要逐段重写。工作流会先理解受众、目标、结构、风格和参考资料，再逐步确认大纲、正文和修改结果。写作过程中产生的有效反馈会立即影响后续内容，经过筛选的经验还可以延续到下一篇文章。

> 作者负责判断写什么，AI 负责完成写作；作者 review，AI 修改并学习。

## 它解决什么问题

AI 写长文最常见的问题，并不是不会写句子，而是协作过程不稳定：

- 作者的要求散落在多轮对话里，越写到后面越容易被忘记。
- AI 没有先理解受众和目标，就直接生成一篇“看起来完整”的文章。
- 参考资料被机械改写，文章没有作者自己的判断和表达。
- 大纲、正文和风格同时调整，问题发现得太晚，返工成本很高。
- 用户已经纠正过的问题，到了下一节、下一篇还要重新说一遍。
- 全文写完以后缺少分层门禁，硬约束、风格、事实和主观判断混在一张评分表里。

Flash Writer 用一份明确的写作思路控制全文，通过分阶段确认降低返工，并用“当前立即生效、长期经过筛选”的双层经验机制保持连续性。

## 核心能力

| 能力 | 作用 |
| --- | --- |
| 写作思路驱动 | 用原则、受众、目标、章节安排和参考资料约束全文 |
| 分阶段确认 | 先对齐信息，再确认大纲，最后逐节写正文 |
| 风格样本学习 | 从作者样本中提取句式、节奏、语气和表达习惯 |
| 参考资料重写 | 借鉴事实和结构，但不复制原文句子 |
| 即时反馈继承 | 用户的新要求从下一段、下一节开始持续执行 |
| 跨文章经验沉淀 | 把真正可复用的规则写入个人写作档案 |
| 项目上下文延续 | 保存需要在同一项目内持续使用的背景，不把项目事实写进个人偏好 |
| 完稿四层自检 | 依次检查硬约束、作者风格、内容事实和作者终审，先修订，再交付待确认项 |
| Token 控制 | 只加载有效规则，不在每次写作时重读全部反馈历史 |

Flash Writer 始终只有这一套工作流，不提供“快速模式”“完整模式”等平行模式。

## 适用场景

适合：

- 公众号文章、博客、专栏和系列文章。
- 教程、产品方法论、案例分析和研究型内容。
- 演讲稿、分享逐字稿和培训材料。
- 已经有写作思路、参考资料或风格样本，需要 AI 协作完成长文。
- 希望在多轮修改中保持要求一致，并逐渐形成个人写作风格。

不适合：

- 只需要一句广告语、一个标题或一段临时文案。
- 不愿意提供受众、目标，也不准备参与大纲和正文确认。
- 希望直接复制或轻度改写他人的文章。

## 安装

将完整的 `flash-writer` 目录放入当前 AI 工具能够发现的 Skill 目录。例如：

```text
~/.agents/skills/flash-writer/
```

不同工具的 Skill 搜索目录可能不同，以当前运行环境的说明为准。不要只复制 `SKILL.md`；`commands/`、`scripts/` 和 `references/` 也是工作流的一部分。

安装后可以直接点名使用 `$flash-writer`。如果当前环境已经注册了 `commands/flash-writer.md`，也可以使用 `/flash-writer` 完成初始化。

## 快速开始

### 方式一：直接提供写作思路文件

如果已经有写作思路，直接告诉 AI：

```text
使用 $flash-writer，严格按照 @写作思路.md 完成这篇文章。
```

AI 会先读取个人有效规则和项目上下文，再校验写作思路、对齐资料并推荐大纲，不会立即生成全文。

### 方式二：从模板开始

1. 复制 [`references/writing-template.md`](references/writing-template.md) 到写作目录。
2. 填写原则、受众、目标、章节安排、风格、参考资料和图片需求。
3. 把文件交给 AI，并要求使用 `$flash-writer`。
4. 确认信息理解和推荐大纲。
5. 按章节协作完成正文。

### 方式三：使用 `/flash-writer`

在支持该命令的环境中输入：

```text
/flash-writer
```

命令会：

1. 询问写作目录。
2. 创建 `references/`，并复制写作模板、完整案例和截图指南。
3. 如果目标目录已经存在 `references/`，改用 `writer-references/`，避免覆盖已有内容。
4. 初始化个人写作档案和项目状态。
5. 等待用户填写写作模板并回复“已写完”。
6. 校验写作思路后进入正式工作流。

## 写作思路应该包含什么

一份可执行的写作思路至少需要回答五个问题：

| 部分 | 要回答的问题 |
| --- | --- |
| 原则 | 这是什么内容？有哪些不能违反的硬约束？ |
| 受众 | 谁会阅读？他们知道什么、困在哪里？ |
| 目标 | 读完以后，读者应该能够做什么或形成什么判断？ |
| 写作思路 | 文章有哪些章节？每节的核心观点、引用资料、希望得出的判断和处理要求是什么？ |
| 内容参考 | 哪些文件或链接提供事实、案例、结构或操作步骤？ |

还可以补充：

- 文章类型和发布平台。
- 标题、表格、段落等格式偏好。
- 一篇或多篇个人风格样本。
- 已有图片、需要 AI 截图的页面或图片占位要求。

### 使用 `》` 标记章节处理方式

在章节下面使用 `》` 写清具体指令：

```text
3、核心案例
》参考 01 资料中的事实，用面向非技术读者的大白话重新讲

4、更多使用场景
》一带而过，列 4—6 个贴近受众工作的例子

5、操作步骤
》参考 02 手册，只保留最短可执行路径
```

`》` 后面的内容是执行指令：

- `参考`：借鉴事实或结构，不能复制表达。
- `一带而过`：简要覆盖，不展开长篇解释。
- `简化`：保留完成目标所必需的内容。
- 没有标注：允许 AI 根据受众和目标提出原创建议。

## 完整工作流

```text
准备写作思路
→ 校验完整性
→ 对齐信息
→ 推荐大纲
→ 多轮确认大纲
→ 逐节写正文
→ 完稿四层自检与修订
→ 确认完稿
→ 沉淀可复用经验
```

### 1. 校验写作思路

检查原则、受众、目标、章节安排和参考资料是否完整，并识别相互冲突的要求。例如，受众是非技术读者，却要求正文直接深入源码，就需要先重新对齐深度。

### 2. 对齐信息

AI 完整读取写作思路，只加载本篇真正需要的约束、参考资料和风格样本，然后向用户汇报：

- 对写作任务的理解。
- 已读取的约束和参考资料。
- 每份资料准备怎样使用。
- 从风格样本中提取的具体特征。
- 仍然存在的歧义或缺口。

用户确认后再进入大纲阶段。

### 3. 推荐并确认大纲

AI 提供 2—3 级大纲，并为每个章节说明：

- 这一节解决什么问题。
- 准备写哪些具体内容。
- 使用哪份参考资料。
- 哪些内容属于 AI 的原创建议。

大纲可以多轮调整。涉及具体对象、工具、案例和顺序时，以用户最后确认的版本为准。

### 4. 逐节写正文

- 第一节完成后先校准风格，尽早发现语气或节奏问题。
- 每个大章节完成后暂停，等待用户 review 或继续；用户不需要亲自重写段落。
- AI 根据 review 重写问题内容，用户反馈会立即进入当前文章的有效要求，后续章节自动复用。
- 参考资料只提供原料，正文必须重新组织和表达。
- 图片、截图和占位符与对应章节一起处理。

### 5. 完稿四层自检和修订

全文完成后，AI 根据本次写作思路、确认后的大纲、当前文章有效要求和个人写作档案生成验收项，按顺序执行四层门禁：

- **L1 硬约束**：检查必需章节、顺序、指定对象、处理指令、篇幅、格式、图片和禁用项；必须零违反。
- **L2 作者风格**：检查语气、句式、节奏、转场和全文一致性，修复模板化、报告化和明显 AI 套话。
- **L3 内容与事实**：检查观点支撑、结构、事实与推断边界、引用依据，以及是否虚构作者经历、数据或对话。
- **L4 作者与读者终审**：检查文章是否真正表达作者观点、是否存在注意力断点，只把最重要的 1-3 个取舍交给作者确认。

AI 能够直接修复的问题先修复并重新检查；缺少依据或会改变作者核心观点时才交给作者。报告只使用“已通过、已修复、待作者确认”，不输出综合分数。

### 6. 显式确认完稿

完成本轮修改后，AI 会展示当前状态、四层自检报告、准备沉淀的经验以及不会长期保存的内容，然后提示：

> 如果你认为文章已经正式定稿，请回复：**确认完稿**。

只有明确回复“确认完稿”，才会锁定最终版本、完成最终自检，并把通过门禁的经验升级为长期个人规则。

提交文件、提供发布链接、开始下一篇或回复“继续”，都不会被自动视为确认完稿。

## 反馈如何自动延续

写作过程中，只要用户的反馈改变了文章要求，AI 会同时完成三件事：

1. 修改当前正文。
2. 在“原始反馈”中记录用户意见和已执行修改。
3. 更新当前文章的“当前有效要求”。

当前有效要求从下一段、下一节开始立即生效，不需要等待完稿。重复规则会合并，发生冲突时只保留最新有效规则。

以下内容不会记录为写作经验：

- “继续”“可以”“下一节”等只推进流程的回复。
- 没有改变文章要求的闲聊。
- 已被后续反馈废止的旧规则。

恢复对话或上下文被压缩后，AI 只加载当前文章的有效要求，不需要重读完整反馈历史。

## 个人经验如何沉淀

长期经验必须同时满足：

- 删除文章专有名词、章节编号和具体对象后仍然成立。
- 可以应用到未来不同主题的文章。
- 描述写作判断、表达偏好或协作方式，而不是本篇修改结果。
- 不依赖会变化的项目事实或外部数据。
- 能写成简短的“适用条件 → 写作动作”规则。

例如：

```text
面向移动端发布 → 避免宽表格，优先使用两列表格或纵向卡片。
```

不会进入个人档案的内容包括：

- 某篇文章的具体标题和章节修改。
- 项目名称、竞品名称、发布地址和当前数据。
- 只在本篇成立的对象范围和事实结论。
- 原始对话、长案例和已经失效的规则。

同一项目需要持续复用、但不能跨项目泛化的事实，进入项目上下文，而不是个人写作档案。

## 状态文件与存储位置

所有个性化状态都保存在当前用户目录，不写入文章目录或代码仓库：

```text
~/.flash-writer/
├── writing-profile.md
└── projects/
    └── <项目名>-<路径哈希>/
        ├── project-context.md
        └── feedback-log.md
```

| 文件 | 保存什么 | 何时加载 |
| --- | --- | --- |
| `writing-profile.md` | 跨文章、跨项目复用的个人写作规则和协作偏好 | 每次开始写作 |
| `project-context.md` | 只在当前项目内延续的有效背景和约束 | 每次开始当前项目的写作 |
| `feedback-log.md` | 每篇文章的当前有效要求、原始反馈和处理结果 | 正常恢复时只读取指定文章的有效要求；整理时才读取完整章节 |

项目识别规则：

- Git 项目使用距离 `--project-dir` 最近的仓库根目录。
- 非 Git 项目使用传入目录。
- 项目名后附路径哈希，避免两个同名目录互相覆盖。
- 个人档案与项目状态不会跟随代码仓库提交。

### 规则优先级

当不同来源的要求发生冲突时，使用以下顺序：

```text
用户当前明确要求
> 当前文章有效要求
> 当前写作思路
> 项目上下文
> 个人写作档案
> Skill 通用规则
```

## Token 消耗控制

为了让个人经验持续增长但不让上下文无限膨胀：

- 开篇只加载个人档案和有效项目上下文。
- 恢复文章时只额外加载该文章的“当前有效要求”。
- 正常写作不读取原始反馈历史。
- 个人档案只保留当前有效的原子化短规则。
- 同义规则及时合并，废止规则留在历史记录中。
- 单个活动区超过 4000 字符时，先压缩再加载。
- 完整读取写作思路，其他资料按章节需要渐进读取。
- 已经获取的 URL 内容保存到本地复用，除非用户要求刷新或信息有明显时效性。

## 图片与外部资料

### 图片

按以下优先级处理：

1. 使用用户提供的截图。
2. 通过浏览器打开真实页面并截图。
3. 插入清楚描述用途的 `TODO` 占位符，等待用户补图。

AI 截图时应使用实际页面、保留与正文相关的状态，并使用描述性文件名。不要从其他作者的文章中提取图片充当原创素材。

### URL

- 只读取当前文章确实需要的链接。
- 获取成功后保存为本地 Markdown 参考资料，避免重复访问。
- 需要登录或无法读取时及时告知用户，不在同一路径反复重试。
- 外部资料提供事实和思路，不提供可直接复制的成稿。

## 状态工具命令

以下命令在 Skill 根目录执行。

### 初始化状态

```bash
python3 scripts/profile_store.py init --project-dir <写作项目目录>
```

只创建缺失文件，不覆盖已有档案。输出项目身份、实际状态路径和新建文件。

### 加载有效状态

```bash
python3 scripts/profile_store.py load --project-dir <写作项目目录>
```

只加载个人有效规则和项目上下文。

恢复指定文章时：

```bash
python3 scripts/profile_store.py load \
  --project-dir <写作项目目录> \
  --article <写作思路文件名或文章标识>
```

这会额外加载指定文章的“当前有效要求”，但不会加载原始反馈历史。

可使用 `--max-chars` 调整单个活动区的字符上限，默认值为 `4000`；使用 `--global-dir` 可以在测试时覆盖默认的 `~/.flash-writer/`。

### 迁移旧版项目内状态

```bash
python3 scripts/profile_store.py migrate \
  --project-dir <写作项目目录> \
  --legacy-dir <旧的项目内 .flash-writer 目录>
```

迁移行为：

- 把 `project-context.md` 和 `feedback-log.md` 移到全局项目状态目录。
- 如果目标位置已经存在不同内容，在移动任何文件前停止。
- 如果源文件与目标文件完全相同，保留目标并删除重复的旧文件。
- 迁移完成后，只删除已经搬空的旧目录。

## 验证

修改 Skill 后至少运行：

```bash
PYTHONPYCACHEPREFIX=/tmp/flash-writer-pycache \
  python3 -m py_compile scripts/profile_store.py
```

然后在一个临时目录中分别验证：

1. `init` 会创建三类状态文件。
2. `load` 不会加载原始反馈历史。
3. `load --article` 只加载目标文章的有效要求。
4. `migrate` 在目标内容冲突时拒绝移动。
5. Skill frontmatter 和目录命名通过当前环境的 Skill 校验器。

## 常见问题

### 只能写系列文章吗？

不能。独立文章可以把项目上下文保持为空；Skill 不会预设所有写作任务都属于系列。

### 需要先把整篇文章想清楚吗？

不需要准备成稿，但必须明确受众、目标和基本章节方向。大纲和表达可以在协作中继续调整。

### 用户反馈什么时候生效？

改变文章要求的反馈从下一段、下一节开始立即生效，不需要等到确认完稿。

### 为什么还要“确认完稿”？

它不是为了让当前修改生效，而是为了明确文章已经锁定，并触发最终自检和长期经验筛选。发布、提交或开始下一篇都不能可靠代表作者已经满意。

### 下一篇会读取全部历史对话吗？

不会。下一篇只加载当前有效的个人规则和项目上下文；原始反馈历史默认不进入上下文。

### 个人写作风格会写进 Skill 吗？

不会。Skill 始终保持通用，个人规则保存在用户自己的 `~/.flash-writer/` 中。

### 会污染代码仓库吗？

不会。个性化状态位于用户目录，`--project-dir` 只用于识别项目身份。

### 可以直接复制参考文章吗？

不可以。参考资料只能用于理解事实、结构和思路，最终内容必须重新组织并使用作者自己的表达。
