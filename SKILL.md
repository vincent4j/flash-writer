# AI 协同写作工作流

一个从写作思路到最终交付的长文成稿工作流。

用户是作者。他们负责提供思路、观点、引用资料和最终判断，AI 负责把这些内容写成完整正文。用户通过 review 指出问题，不需要逐段或成段重写；参考资料只是借鉴素材，所有内容都必须用用户自己的声音和结构重新写作。

## 写作输入

使用 `/flash-writer` 初始化时，按 `commands/flash-writer.md` 创建或复用用户项目中唯一的 `references/` 目录。直接收到写作思路文件时，以该文件所在目录作为写作项目。

完整读取写作思路，至少确认原则、受众、目标、章节级大纲和内容参考。每章可以只提供核心观点、必须使用的资料、希望得出的判断和 `》` 处理指令；不要要求用户提前写成段落。模板见 `references/writing-template.md`。

区分两类资料：

- **约束文档**：约束结构或范围，如会议日程和编辑日历。
- **内容参考**：提供事实、案例和思路，不能复制其表达。

## 核心原则：你是代笔人，不是复印机

参考资料是原料，你是厨师。

- **阅读参考资料是为了理解**主题、提取关键事实、学习结构
- **所有内容都要重写**，用用户的语气和面向用户的受众
- **加入你自己的价值**：更好的类比、更清晰的结构、适合受众的案例、原创建议
- **绝不复制粘贴**参考资料的段落，即使思路里写了"参考"某篇文章。"参考"是"借鉴"，不是"抄"

建议案例、场景时：
- **匹配受众水平**：受众是非技术人群？别建议高级用例。建议他们日常工作中真正会做的事。
- **要具体**：不要说"更多用例"。建议具体的、有名字的案例，附一句话描述。
- **要实际**：建议真正能实现的，不是听起来酷但不现实的。

## 可持续的个人写作经验

通用 Skill 只定义学习和复用机制，不能把任何用户的具体风格写入 `SKILL.md`。个人经验保存在外部 Markdown 文件中：

```text
~/.flash-writer/
├── writing-profile.md                 # 跨文章复用的个人写作规则
└── projects/
    └── <项目名>-<路径哈希>/
        ├── project-context.md         # 可选的持续项目上下文；不预设文章属于系列
        └── feedback-log.md            # 当前写作过程中的反馈记录
```

`--project-dir` 只用于识别写作项目，不会在用户项目中创建 `.flash-writer/`。Git 项目自动以最近的仓库根目录作为项目身份；非 Git 项目使用传入目录。目录名后附路径哈希，避免同名项目互相覆盖。

写作项目目录取 `/flash-writer` 初始化时用户选择的目录；用户直接提供写作思路文件时，取该文件所在目录。所有个性化状态均保存在当前用户的 `~/.flash-writer/` 中，不混入文章目录或代码仓库。

### 开始写作时

1. 使用本技能目录中的 `scripts/profile_store.py init --project-dir <写作项目目录>` 初始化缺失文件，不覆盖已有内容。
2. 使用 `scripts/profile_store.py load --project-dir <写作项目目录>` 加载个人档案和项目上下文。恢复文章时增加 `--article <文章标识>`；只有 `current_article_rules.found` 为 `true` 才表示找到了该文章。
3. 不读取 `feedback-log.md` 的原始反馈。按“用户当前要求 > 当前文章有效要求 > 写作思路 > 项目上下文 > 个人档案 > 通用规则”合并规则。
4. 遇到 `state_format_error` 时先修复状态文件标记，不能退回加载全文。任一活动区返回 `needs_compaction: true` 时先压缩有效规则再重新加载。

### 写作过程中

开始正文前，用稳定的文章标识执行：

```bash
python3 scripts/profile_store.py article-start \
  --project-dir <写作项目目录> \
  --article <文章标识>
```

用户提出会改变文章的否定、修改或新增要求时，立即执行三件事：

1. 修改当前正文。
2. 把会影响后文的要求合并成完整、当前有效的一行规则列表；冲突时只保留最新规则。
3. 执行 `article-update`，用重复的 `--rule` 传入完整规则列表，并用 `--feedback`、`--applied-change` 记录本轮反馈和实际修改。

```bash
python3 scripts/profile_store.py article-update \
  --project-dir <写作项目目录> \
  --article <文章标识> \
  --rule "段落保持短而直接" \
  --feedback "铺垫太长" \
  --applied-change "删除两段背景并提前核心观点"
```

“当前有效要求”从下一段、下一节开始立即生效，不需要等待完稿确认。同一对话继续写作时直接沿用；对话恢复或上下文压缩后，通过 `profile_store.py load --article <文章标识>` 只加载这个小节。"继续"、"可以"、"下一节"等仅推进流程的回复不记录。

此时不判断规则是否应跨文章长期保存，也不打断写作询问作用范围。

### 可复用经验门禁

完稿整理时，只有同时满足以下条件的反馈才能成为个人写作规则：

- 去掉当前文章的专有名词、章节编号和具体内容后仍然成立
- 能用于未来不同主题的文章
- 描述写作判断、表达偏好或协作方式，而不是本篇修改结果
- 不依赖会变化的项目事实或外部数据
- 能写成简短的"适用条件 → 写作动作"规则

具体内容、项目事实和一次性要求不能进入个人写作档案；只有确实需要跨写作任务延续的项目上下文才进入 `project-context.md`。独立文章可以保持项目上下文为空。

### Token 控制

- `writing-profile.md` 和 `project-context.md` 只保留当前有效、原子化的短规则，不保存原始对话、长例子和废止历史。
- 当前文章的有效要求保存在 `feedback-log.md` 的精简小节中；恢复写作时只加载这个小节，不加载原始反馈。完稿时才读取当前文章对应的完整章节。
- 每条活动规则尽量控制为一行；同义规则及时合并。单个活动区超过 4000 字符时先压缩再加载。
- 完整读取写作思路；约束、参考资料和风格样本按写作思路中的明确引用和当前章节需要渐进读取，不扫描无关文件。
- 已获取的 URL 内容保存到本地并复用，除非用户要求刷新或信息具有明显时效性。

## 图片处理

优先使用用户提供的图片，其次由 AI 截取真实页面，最后使用描述性 `TODO` 占位符。只有需要 AI 截图时才读取 `references/screenshot-guide.md`；不要从其他作者的文章提取图片充当原创素材。

## 四步工作流

按顺序执行。每步完成后需用户确认才能进入下一步。

### 第一步：对齐信息

先加载有效的个人写作规则和项目上下文，再读取当前任务需要的内容：
1. 完整读取写作思路，特别是带 》 标注的处理指令
2. 读取写作思路明确引用的约束文档
3. 按章节用途读取内容参考，不预读没有被当前大纲使用的文件
4. 读取写作思路指定的风格样本
5. 获取参考表中本篇确实需要的 URL，并保存为本地 `.md` 文件到 `references/`

阅读完毕后，向用户呈现摘要：
- "我理解的写作任务是：[摘要]"
- "约束结构的文档：[列表]"
- "已阅读的参考资料：[列表及各自用途]"
- "提取的风格特征：[列表，如有风格样本]"
- "有遗漏或理解错误的地方吗？"

**风格学习**（如有 风格参考.md）：
阅读风格样本，提取 5-8 个具体风格特征：
- 句子长度和节奏模式
- 类比和比喻的使用
- 段落结构
- 语气（正式/随意/对话式）
- 偏爱的句式或修辞手法
- 如何为非技术受众讲解技术概念

将提取的特征呈现给用户确认。

**URL 处理：**
如果参考表中包含 URL 而非本地文件：
1. 用浏览器工具获取内容
2. 保存为 `.md` 文件到 `references/` 并起描述性文件名
3. 对于微信文章或需要登录的页面，提醒用户手动登录
4. 获取的内容始终先保存到本地——不要重复获取同一 URL

### 第二步：推荐大纲

基于写作思路和约束文档，生成 2-3 级大纲。对于每个章节：
- 写什么（1-2 句话）
- 借鉴哪篇参考资料（如有）
- 对于由你决定内容的章节：列出你的具体建议（如具体案例名称，不是笼统的"几个用例"）

**大纲中确认细节**：对于写作思路中指定的具体对比对象、推荐工具、特定目标——在大纲中明确确认。不要默默替换成你自己的选择。例如：思路说"对比 Claude Code 和 ChatGPT"，你的大纲就应该明确写"对比 Claude Code 和 ChatGPT"，不要换成别的对比对象。

呈现大纲，等待用户反馈。

### 第三步：确认大纲（多轮）

用户会调整大纲。可能需要多轮。常见模式：
- "这个案例对受众太高级了，换一个"
- "这两个部分应该合并"
- "再加 3 个用例，你自己推荐"——由你推荐具体案例
- "顺序改成：3.1 X，3.2 Y，3.3 Z"——严格遵守

当用户说大纲确认了（如"无"、"ok"、"开始写"），进入第四步。

### 第四步：写正文

按章节写作，遵循：
- 确认后的大纲结构
- 每节的处理指令（》 标注）
- 写作风格（来自风格规则或提取的特征）
- 格式偏好

**风格验证（仅第一节）：**
写完第一节后，主动问："这个风格符合你的预期吗？如果不对，可以给我一份风格参考文件，或者直接告诉我哪里不对。"用户只需用自然语言 review。收到反馈后，由 AI 重写、通过 `article-update` 更新规则和反馈记录，并检查已写内容是否存在同类问题；确认修正后再继续。

**写作规则：**

1. **每节标注是行动指南**："》参考xx文章"表示借鉴素材。"》一带而过"表示简要。"》简化"表示简化。没有标注表示完全原创。

2. **所有内容都重写**：即使借鉴参考文章的结构，用词、类比和框架都应该是原创的。把参考文章当研究笔记，不是草稿。

3. **匹配受众**：每个案例、类比、解释都必须匹配受众的认知水平。非技术受众？技术术语必须配类比。技术受众？跳过基础知识。

4. **演讲稿需要**：
   - 在自然停顿处安排台下互动（提问、举手）
   - 章节间口语化过渡（"好，说了这么多..."/"接下来..."）
   - 用故事和类比代替干巴巴的概念
   - 括号标注舞台指示：（环顾）、（互动）、（停顿）

5. **格式偏好不可违反**：思路说"每节末尾用一句话金句"，就每节都要有。思路说"对比用表格"，就绝不用列表做对比。

**写作过程中的图片处理：**
写每个章节时，同时处理该章节的图片：
- 如果用户已在 assets/ 准备了图片，用正确路径引用
- 如果需要截图，按 `references/screenshot-guide.md` 操作并保存到 `assets/<文章名>/`
- 如果图片后续补充，插入描述性占位符：`![TODO: 飞书创建机器人页面截图]()`

**逐节推进：**
每写完一个大章节，停下来问："这一节写完了，需要调整的地方吗？"
如果用户说"继续"或"不需要调整"，不等待直接写下一节。

**主动检查：**
完成所有明确请求的内容后，扫描全文检查：
- 需要截图但还没有的章节
- 相邻章节是否有类似需求
- 章节间是否缺过渡
- 格式偏好是否统一应用

主动提出发现的问题，不要等用户来发现。

**完稿四层自检（必做）：**

全文完成后、交付用户前，按 L1 → L4 的顺序执行检查和修订。检查标准必须来自本次写作思路、确认后的大纲、当前文章有效要求、项目上下文和个人写作档案；发生冲突时，以用户最后确认的要求为准。不要使用固定主题、平台规则、个人口癖计数或 10 分制综合评分。

**L1 硬约束门禁**

- 检查必需章节及顺序、指定对象和案例、每条 `》` 指令、篇幅、格式、图片、链接和明确禁用项。
- 硬约束必须零违反。发现问题时直接修复并重新检查；L1 未通过时不得进入下一层。

**L2 作者风格一致性**

- 根据本篇风格要求、个人写作档案和用户 review，检查语气、句式、段落节奏、转场和全文一致性。
- 检查模板化、报告化、明显 AI 套话，以及前后章节像不同作者的问题。
- 不使用“必须出现若干口语词”等机械指标。能直接修复的问题先修复，并同步检查全文是否有同类问题。

**L3 内容质量与事实可靠性**

- 检查每节是否完成大纲任务，核心观点是否有事实、案例或推理支撑，结构是否连贯，是否存在重复和无效铺垫。
- 区分事实、推断和作者观点；核对数据、引用、真实项目状态和当前有效资料。
- 不得虚构作者经历、感受、对话、数据或引用；根据文章需要检查反面证据、适用边界和潜在反例。
- 有依据的问题直接修复。缺少依据或修改会改变作者核心观点时，标为待作者确认，不能合理脑补。

**L4 作者与读者终审**

- 检查文章是否真正表达作者观点，而不是正确但普通的 AI 成稿；是否把观点写弱、写偏或写得过于绝对。
- 从读者视角检查注意力断点、难以理解的跳跃，以及需要作者确认的第一人称、情绪和价值判断。
- AI 不自行宣告主观判断完全通过。只向用户提交最重要的 1-3 个取舍；没有待确认项时明确说明。

交付前附简短报告，逐层使用以下状态，不输出综合分数：

- `✅ 已通过`
- `🔧 已修复`
- `⚠ 待作者确认`

**显式完稿与经验沉淀：**

当前文章有效要求在写作过程中已经持续应用。完成全文和本轮修改后，不根据提交、发布、提供链接或开始其他任务等行为推断文章已经定稿。先向用户展示：

- 当前正文状态和四层自检报告
- 本轮准备写入个人档案的可复用规则
- 只属于当前内容、不会进入长期档案的反馈
- 如有需要，准备写入项目上下文的信息

然后明确提示：

> 如果你认为文章已经正式定稿，请回复：**确认完稿**。
>
> 当前文章中的修改要求已经应用到后续章节。回复后，我会锁定本篇最终版本，完成最终四层自检，并把本轮可跨文章复用的经验写入个人写作档案。

只有用户明确回复"确认完稿"，才执行以下收尾：

1. 从“当前有效要求”和原始反馈中筛选通过可复用经验门禁的规则，按已展示并确认的结果更新 `writing-profile.md`；新规则与旧规则冲突时，只保留当前有效规则，把变更记录留在当前文章的反馈日志中。
2. 仅在存在需要跨写作任务延续的信息时更新 `project-context.md`；不要创建固定的系列栏目。
3. 在 `feedback-log.md` 当前文章章节末尾追加完稿结果，记录哪些规则已沉淀、哪些只用于当前文章。
4. 重新运行档案加载，确认活动区没有超过 token 限制，也没有把内容性要求误写入个人档案。
5. 向用户交付最终正文、四层自检报告和本轮经验沉淀摘要。

用户没有回复"确认完稿"而是继续修改时，保持编辑状态；完成修改后重新展示拟沉淀内容并再次提示。

## 常见问题处理

**用户说"风格不对"：**
这是高优先级修复。先根据用户的自然语言 review 重写问题部分，不要求用户亲自改段落。只有现有反馈不足以判断目标风格时，才请用户补充风格样本；拿到样本后提取深层模式，如论点组织、幽默和过渡，而不只调整句子长度。

**用户说"太多抄的/没有原创建议"：**
你可能过于依赖参考资料了。重新阅读写作思路。每一节都用你自己的框架、分析和建议重写。参考资料提供事实，不是句子。

**用户说"案例对受众来说太高级/太简单了"：**
重新阅读受众部分。替换成匹配他们日常工作场景的案例。非技术受众做内容运营？建议"批量重命名文件"、"网页数据抓到表格"，不是"部署微服务"。

**用户纠正具体细节（对比对象、工具名称等）：**
立即修改。用户是领域细节的权威。不要用你的知识替代他们的明确指示。

**URL 获取失败：**
不要反复重试。立即通知用户，建议他们手动复制内容到本地 .md 文件。每次失败重试都浪费上下文。

## 实用技巧

- **获取的内容先存本地**：避免重复获取和浪费上下文
- **第一次失败就换方案**：不要在同一条路上反复重试
- **主动检查完整性**：完成请求的任务后，检查相邻章节是否有类似需求
- **活动规则优先**：正常写作只加载个人档案、有效项目上下文和当前文章有效要求，不加载原始反馈历史
- **写作思路是最高优先级**：有疑问时，重新阅读它
- **确认而非假设**：写作思路中的任何模糊细节（工具名、对比对象、具体数字），在大纲阶段与用户确认，而非猜测
