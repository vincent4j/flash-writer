# AI 协同写作工作流

一个结构化的长文协作写作工作流。用户准备一份写作思路文件（模板），填好所有要求，本技能引导从信息对齐到最终交付的完整写作过程。

用户是作者。他们清楚知道自己要写什么。参考资料只是借鉴素材——所有内容都必须用用户自己的声音和结构重新写作。

## 触发条件

- 用户通过 `/flash-writer` 命令启动（走初始化流程：创建文件 → 填写模板 → 校验 → 进入四步工作流）
- 用户指向写作思路/模板文件："@xxx.md 按这个要求执行"、"@brief.md 按这个写"
- 用户要求写一篇文章、演讲稿、教程等结构化内容，并有参考资料
- 用户有一个包含参考文件的目录，想产出一篇成品

## 目录结构

触发时，应有（或帮助创建）如下结构：

```
flash-writer/
├── SKILL.md                        # 主技能文件（规则 + 工作流）
├── CHANGELOG.md                    # 更新日志
├── commands/
│   └── flash-writer.md             # /flash-writer 命令定义
├── references/
│   ├── writing-template.md         # 可复制的写作模板
│   ├── real-example.md             # 完整实际案例
│   └── screenshot-guide.md         # 截图最佳实践
└── worklog/                        # 每日工作日志和复盘
```

注意区分：
- **约束文档/**：定义你必须遵循的结构的文档（如会议日程、编辑日历）。它们约束大纲，不是内容来源。
- **参考资料/**：借鉴思路、事实和结构的内容来源。但你始终重写——不复制粘贴。

## 写作模板格式

写作思路文件应包含以下部分。如有缺失，在开始前让用户补充。可复制的模板在 `references/writing-template.md`。

### 必填部分

**原则：** 这是什么类型的内容？硬约束是什么？（如"演讲稿需要口语化"、"教程必须可操作"）。如有约束文档（会议日程等），在这里引用。

**受众：** 谁在看/听？他们当前的认知水平和痛点是什么。这一部分至关重要——它决定了你写的每一段的语言复杂度、案例选择和深度。

**目标：** 读者看完/听完之后应该达到什么状态？用具体场景描述，不要用抽象的"了解 X"。

**写作思路：** 章节级大纲，含每章的处理指令。特别注意 》 标记：

```
3、案例：24 小时全自动热点收割机
》参考 01 文章的案例，用大白话重新讲

4、更多应用场景展示
》一带而过，重数量（4-8 个），贴近受众

5、安装指南
》参考 02 文章，简化步骤，只保留最短路径
```

》 符号标记每节的处理指令，是你最重要的行动指南：
- "参考xx文章" = 借鉴该参考资料的结构/事实，但用你自己的话重写
- "一带而过" = 简要覆盖，不深入
- "简化" = 简化，去掉不必要的复杂性
- 没有标注 = 由你决定内容和写法，提供你自己的建议

**内容参考：** 参考来源表：

| 文件/链接 | 用途 | 备注 |
|----------|------|------|
| `参考资料/01-xxx.md` 或 URL | 案例部分的主要参考 | 借鉴案例结构，用大白话重写 |
| `参考资料/02-yyy.md` 或 URL | 安装部分的参考 | 以步骤为基础，简化 |

### 选填部分

**文章类型：** 文章、演讲稿/逐字稿、技术教程、产品介绍等。演讲稿需要设计台下互动和口语化过渡。

**格式偏好：** 如：标题不加序号、对比内容用表格、每个模块末尾用一句话金句收尾、标题用 emoji 装饰。

**写作风格：** 直接描述风格规则（如"口语化，段落短，喜欢用类比"），或指向风格样本文件。

**图片素材：** 图片处理方式：
- 用户提前准备好截图放在 `assets/` 目录
- AI 写作时用浏览器工具截图
- 用户后续补充图片（插入 `![TODO: 描述]()` 占位符）

## 核心原则：你是代笔人，不是复印机

参考资料是原料，你是厨师。

- **阅读参考资料是为了理解**主题、提取关键事实、学习结构
- **所有内容都要重写**，用用户的语气和面向用户的受众
- **加入你自己的价值**：更好的类比、更清晰的结构、适合受众的案例、原创建议
- **绝不复制粘贴**参考资料的段落，即使思路里写了"参考"某篇文章。"参考"是"借鉴"，不是"抄"

建议案例、场景时：
- **匹配受众水平**：受众是非技术人群？别建议高级用例。建议他们日常工作中真正会做的事。
- **要具体**：不要说"更多用例"。建议具体的、有名字的案例，附一句话描述。
- **要实际**：建议真正能实现的，不是听起来酷但不现实的。

## 图片处理

文章中的图片应该是用户自己截的或你用浏览器工具截的。不要从别人的文章提取或下载图片。

**方式（按优先级）：**
1. **用户提供截图**：写作开始前准备好图片放在 `assets/`
2. **AI 截图**：写作过程中，用浏览器工具导航到相关页面截图，保存到 `assets/<文章名>/`
3. **占位符模式**：插入 `![TODO: 需要截的页面描述]()` 让用户后续补充

自己截图时：
- 导航到讨论中的实际产品/工具/页面
- 截取相关 UI 状态的干净截图
- 用描述性文件名保存（如 `飞书-创建应用.png`，不是 `截图-1.png`）
- 在文章中引用为 `![](assets/<文章名>/文件名.png)`

## 四步工作流

按顺序执行。每步完成后需用户确认才能进入下一步。

### 第一步：对齐信息

阅读项目目录中的所有内容：
1. 写作思路（每个部分，特别是带 》 标注的写作思路）
2. 约束文档（会议日程等）——定义结构
3. 参考资料/ 目录中的所有文件——提供素材
4. 风格样本文件（如有）
5. 参考表中列出的 URL——用浏览器工具获取并保存为本地 .md 文件到 参考资料/

阅读完毕后，向用户呈现摘要：
- "我理解的写作任务是：[摘要]"
- "约束结构的文档：[列表]"
- "已阅读的参考资料：[列表及各自用途]"
- "提取的风格特征：[列表，如有风格样本]"
- "有遗漏或理解错误的地方吗？"

**风格学习**（如有 风格参考.md）：
阅读风格样本，提取 5-8 个具体风格特征：
- 句子长度和节奏模式
- 类比和比喻的使用
- 段落结构
- 语气（正式/随意/对话式）
- 偏爱的句式或修辞手法
- 如何为非技术受众讲解技术概念

将提取的特征呈现给用户确认。

**URL 处理：**
如果参考表中包含 URL 而非本地文件：
1. 用浏览器工具获取内容
2. 保存为 .md 文件到 参考资料/ 并起描述性文件名
3. 对于微信文章或需要登录的页面，提醒用户手动登录
4. 获取的内容始终先保存到本地——不要重复获取同一 URL

### 第二步：推荐大纲

基于写作思路和约束文档，生成 2-3 级大纲。对于每个章节：
- 写什么（1-2 句话）
- 借鉴哪篇参考资料（如有）
- 对于由你决定内容的章节：列出你的具体建议（如具体案例名称，不是笼统的"几个用例"）

**大纲中确认细节**：对于写作思路中指定的具体对比对象、推荐工具、特定目标——在大纲中明确确认。不要默默替换成你自己的选择。例如：思路说"对比 Claude Code 和 ChatGPT"，你的大纲就应该明确写"对比 Claude Code 和 ChatGPT"，不要换成别的对比对象。

呈现大纲，等待用户反馈。

### 第三步：确认大纲（多轮）

用户会调整大纲。可能需要多轮。常见模式：
- "这个案例对受众太高级了，换一个"
- "这两个部分应该合并"
- "再加 3 个用例，你自己推荐"——由你推荐具体案例
- "顺序改成：3.1 X，3.2 Y，3.3 Z"——严格遵守

当用户说大纲确认了（如"无"、"ok"、"开始写"），进入第四步。

### 第四步：写正文

按章节写作，遵循：
- 确认后的大纲结构
- 每节的处理指令（》 标注）
- 写作风格（来自风格规则或提取的特征）
- 格式偏好

**风格验证（仅第一节）：**
写完第一节后，主动问："这个风格符合你的预期吗？如果不对，可以给我一份风格参考文件，或者告诉我哪里不对。"在写其余部分前修正风格问题——越早发现越省事。

**写作规则：**

1. **每节标注是行动指南**："》参考xx文章"表示借鉴素材。"》一带而过"表示简要。"》简化"表示简化。没有标注表示完全原创。

2. **所有内容都重写**：即使借鉴参考文章的结构，用词、类比和框架都应该是原创的。把参考文章当研究笔记，不是草稿。

3. **匹配受众**：每个案例、类比、解释都必须匹配受众的认知水平。非技术受众？技术术语必须配类比。技术受众？跳过基础知识。

4. **演讲稿需要**：
   - 在自然停顿处安排台下互动（提问、举手）
   - 章节间口语化过渡（"好，说了这么多..."/"接下来..."）
   - 用故事和类比代替干巴巴的概念
   - 括号标注舞台指示：（环顾）、（互动）、（停顿）

5. **格式偏好不可违反**：思路说"每节末尾用一句话金句"，就每节都要有。思路说"对比用表格"，就绝不用列表做对比。

**写作过程中的图片处理：**
写每个章节时，同时处理该章节的图片：
- 如果用户已在 assets/ 准备了图片，用正确路径引用
- 如果需要截图，阅读 `references/screenshot-guide.md` 了解最佳实践，然后用浏览器工具截图保存到 `assets/<文章名>/`
- 如果图片后续补充，插入描述性占位符：`![TODO: 飞书创建机器人页面截图]()`

**逐节推进：**
每写完一个大章节，停下来问："这一节写完了，需要调整的地方吗？"
如果用户说"继续"或"不需要调整"，不等待直接写下一节。

**主动检查：**
完成所有明确请求的内容后，扫描全文检查：
- 需要截图但还没有的章节
- 相邻章节是否有类似需求
- 章节间是否缺过渡
- 格式偏好是否统一应用

主动提出发现的问题，不要等用户来发现。

## 常见问题处理

**用户说"风格不对"：**
这是高优先级修复。让用户提供风格样本文件（风格参考.md）。阅读后重新提取特征，重写问题部分。不要只调整表面的东西（如句子长度），要看更深层的模式——用户如何组织论点、使用幽默、处理过渡。

**用户说"太多抄的/没有原创建议"：**
你可能过于依赖参考资料了。重新阅读写作思路。每一节都用你自己的框架、分析和建议重写。参考资料提供事实，不是句子。

**用户说"案例对受众来说太高级/太简单了"：**
重新阅读受众部分。替换成匹配他们日常工作场景的案例。非技术受众做内容运营？建议"批量重命名文件"、"网页数据抓到表格"，不是"部署微服务"。

**用户纠正具体细节（对比对象、工具名称等）：**
立即修改。用户是领域细节的权威。不要用你的知识替代他们的明确指示。

**URL 获取失败：**
不要反复重试。立即通知用户，建议他们手动复制内容到本地 .md 文件。每次失败重试都浪费上下文。

## 实用技巧

- **获取的内容先存本地**：避免重复获取和浪费上下文
- **第一次失败就换方案**：不要在同一条路上反复重试
- **主动检查完整性**：完成请求的任务后，检查相邻章节是否有类似需求
- **写作思路是最高优先级**：有疑问时，重新阅读它
- **确认而非假设**：写作思路中的任何模糊细节（工具名、对比对象、具体数字），在大纲阶段与用户确认，而非猜测
