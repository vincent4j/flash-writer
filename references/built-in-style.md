# Flash Writer 内置默认写作风格

这份预设提炼自作者已经确认的个人写作风格。只迁移表达方式、组织节奏和判断习惯，不复用任何样本文章的主题、项目名称、固定开场、事实、案例或原句。

## 使用边界

- 写作思路没有填写风格要求，也没有提供风格样本时，自动使用本预设。
- 用户当前要求、当前文章有效要求、写作思路中的风格说明或样本高于本预设。
- 个人写作档案中已经确认的规则高于本预设。
- 冲突时执行高优先级要求；不冲突的规则可以组合使用。

## 默认风格

### 从真实问题进入，尽快给出判断

从真实任务、具体困境、读者常见疑问或正在发生的选择进入正文。先让读者知道问题是什么，再尽快说出作者的判断；不要用长篇行业背景、概念定义或悬念拖延结论。

### 像本人在旁边讲

使用自然口语和短段落，多数段落保持一到三句话。允许真实情绪、随口评论和生活化细节，但不刻意复制口头禅，也不保留病句、错别字或事实错误。

### 答案先行，再把答案讲透

先告诉读者有哪些选择、作者更推荐什么或核心结论是什么，再用问题、证据、案例、原理和实操逐层解释。呈现顺序可以答案先行，但事实因果必须真实，不能先定答案再拼理由。

### 允许有功能的重复

同一判断可以依次出现为一句话答案、直白解释、对比、步骤和结尾压缩，但每次都必须降低理解成本或服务不同阅读耐心。删除没有新增信息的同义改写和机械总结。

### 有个人判断，也有事实边界

面对多个方案时，明确说出作者会选什么、为什么选、哪些方案不适合当前场景。强判断只用于作者的实践结论和选择；外部事实、数据和他人能力必须区分事实、推断和观点。

### 方法论藏在真实过程里

用“真实问题 → 作者判断 → 分析与修正 → 方法点明 → 可用产物”组织内容。方法论是骨架，不要把正文写成“定义—输入—步骤—输出”的考试答案。

### 保留真实过程，不虚构经历

教程和案例可以展示真实指令、第一次结果、发现的问题和后续修正。没有发生过的经历、对话、失败、顿悟和数据不能为了故事感而编造；需要反例时优先使用通用场景。

### 标题直接，Emoji 只服务正文

文章标题和章节标题使用直白、口语化、能说明主题的表达，不加 Emoji。正文只在提问、转折、提醒、关键发现、状态判断、列表或真实情绪变化时使用 Emoji；不建立固定映射，也不机械铺满每段。

### 面向移动端控制结构和篇幅

优先删除无效背景、重复判断和不影响结论的解释，同时保留方法、证据和关键过程。避免宽表格；优先使用两列表格、分组表格、步骤或纵向结构。

### 结尾给判断和下一步

不要只重复全文。结尾压缩最重要的判断，给出读者能够立即完成的动作；系列内容再补充上一篇、下一篇或目录入口。

## 写完后的风格检查

- 正文是否在较短篇幅内出现真实问题和明确判断？
- 是否像作者本人在讲，而不是品牌稿、报告或教材？
- 个人判断是否有力量，外部事实是否保持严谨？
- 方法论是否通过真实问题和证据展开？
- 是否虚构作者经历、情绪、对话、数据或试错？
- 标题是否没有 Emoji，正文 Emoji 是否确实承担信息或情绪作用？
- 是否适合移动端阅读，且没有无效铺垫、宽表格和机械总结？
